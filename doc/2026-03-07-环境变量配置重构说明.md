# 2026-03-07 环境变量配置重构说明

## 1. 本次改动的背景

本次改动的目标，是把项目中原本分散、隐式、缺少校验的环境变量读取方式，重构为一套更适合长期维护的配置体系。

重构前的主要问题有：

- `server.js` 直接在入口手动加载 `.env`，其余业务文件再各自读取 `process.env`
- JWT、CORS、Cookie、数据库配置分散在多个文件中，配置来源不统一
- 数据库连接信息曾经直接写在源码里，存在明显的安全风险
- 配置缺失时不会在启动期失败，而是可能在第一次请求时才暴露问题
- 没有统一的配置模板文件，不利于新环境部署和团队协作

这类问题在项目规模小的时候还能容忍，但随着环境数量增加、部署方式变化、配置项增多，维护成本会迅速上升。

---

## 2. 本次改动的设计目标

这次重构遵循了以下原则：

- 单一入口：只保留一个配置中心负责读取 `process.env`
- 启动即校验：关键配置缺失时直接启动失败
- 配置只读：业务代码只能消费配置，不能修改配置
- 环境无关：本地、测试、生产通过环境变量切换，不通过改源码切换
- 降低迁移成本：尽量保留原有 `require('./config')` 的调用方式
- 避免明文凭据：数据库信息和密钥不再写死在源码中

---

## 3. 本次新增和修改的核心文件

### 3.1 `config/index.js`

这是这次重构的核心文件，职责如下：

- 加载 `.env`
- 加载 `.env.local`
- 对环境变量做类型解析
- 对必填项做启动期校验
- 导出结构化配置对象

当前输出的配置结构主要分为：

- `config.app`
- `config.auth`
- `config.cookie`
- `config.cors`
- `config.db`

这样做之后，业务代码不再直接访问 `process.env.JWT_SECRET`、`process.env.PORT`、`process.env.DB_HOST` 这类原始变量，而是统一改为：

- `config.app.port`
- `config.auth.jwtSecret`
- `config.cookie.secure`
- `config.cors.origins`
- `config.db.host`

这样做的优点是：

- 默认值集中管理
- 校验逻辑集中管理
- 配置来源清晰
- 后续重命名变量时影响面更可控

### 3.2 `config.js`

这个文件现在变成了一个兼容层，只做一件事：

- 把旧的 `require('./config')` 转发到新的 `config/index.js`

保留这个兼容层的原因是，项目里已经有多个模块依赖 `../config` 或 `./config`。如果直接删除，会带来不必要的大面积引用改动。通过兼容层，可以在较低风险下完成重构。

### 3.3 `.env.example`

新增这个文件，是为了提供一份标准配置模板。

它的价值在于：

- 新环境初始化时可以直接对照填写
- 部署时可以清楚知道哪些变量必须配置
- 团队协作时不需要口头同步变量名
- 避免把真实凭据提交到仓库

### 3.4 `CHANGELOG.md`

新增标准化 changelog，采用 Keep a Changelog 格式。

作用是：

- 记录本次改动范围
- 给后续版本追踪提供结构化历史
- 方便排查某项行为是在哪次改动引入的

### 3.5 `doc/`

新增 `doc/` 目录，用于存放“按日期命名的详细修改说明”。

这个目录和 changelog 的定位不同：

- `CHANGELOG.md`：偏版本摘要，写“改了什么”
- `doc/日期-说明.md`：偏技术记录，写“为什么改、怎么改、涉及哪些文件、后续怎么维护”

后续建议每次涉及中大型改动时，都同步补一份日期文档。

## 4. 各业务文件的调整说明

### 4.1 `server.js`

改动前：

- 在入口直接执行 `dotenv.config(...)`
- 直接读取 `process.env.FRONTEND_URL`
- 直接读取 `process.env.PORT`

改动后：

- 不再负责加载 `.env`
- 改为直接依赖统一配置对象 `config`
- CORS 使用 `config.cors.origins`
- 端口使用 `config.app.port`

这意味着 `server.js` 的职责被收缩为：

- 组装应用
- 注册中间件
- 注册路由
- 启动服务

而不再兼顾配置解析工作。

### 4.2 `db/index.js`

改动前：

- 数据库配置通过旧 `config.js` 传入
- 旧配置中包含硬编码数据库凭据

改动后：

- 数据库配置从统一配置中心读取
- 配置项包括 `host`、`port`、`user`、`password`、`database`
- 连接池参数也统一归口

这样做后，数据库相关配置可以通过环境变量安全切换，不需要修改源码。

### 4.3 `services/authService.js`

改动前：

- 直接读取 `process.env.JWT_SECRET`
- 直接读取 `process.env.JWT_REFRESH_SECRET`
- 直接读取 token 过期时间变量

改动后：

- 改为统一读取 `config.auth`
- 登录、刷新 token 的配置来源完全一致

这解决了一个常见风险：不同文件从不同地方读取密钥或过期时间，最终导致生成和校验策略不一致。

### 4.4 `middleware/authMiddleware.js`

改动前：

- 直接用 `process.env.JWT_SECRET` 验证 token

改动后：

- 改为用 `config.auth.jwtSecret`

这样做的意义不仅是“代码更整洁”，更重要的是：

- 如果 `JWT_SECRET` 缺失，应用会在启动期就失败
- 不会等第一条受保护请求打进来时才发现鉴权配置错误

### 4.5 `controllers/authController.js`

改动前：

- `secure` 直接由 `NODE_ENV === 'production'` 决定
- `sameSite`、`maxAge` 写死在控制器

改动后：

- Cookie 策略统一从 `config.cookie` 读取
- 支持按配置决定 `secure`
- 支持通过配置控制 `sameSite`
- 支持可选的 `domain`

这样做的好处是：

- 本地开发、测试环境、生产环境可以使用不同 Cookie 策略
- 控制器只负责响应逻辑，不负责环境判断逻辑

## 5. 配置加载和优先级说明

当前配置加载逻辑如下：

1. 先读取进程启动时已有的环境变量
2. 再尝试加载项目根目录 `.env`
3. 再尝试加载项目根目录 `.env.local`
4. 如果某个变量在系统环境里已经存在，则本地文件不覆盖它

这套优先级的意义是：

- 部署系统注入的正式配置优先级最高
- 本地开发可以通过 `.env` 提供基础配置
- 开发者个人机器可以通过 `.env.local` 做个性化覆盖

这比单纯无脑 `dotenv.config()` 更接近生产可维护模式。

## 6. 启动期校验说明

本次新增了一组配置解析函数：

- `getRequired`
- `getString`
- `getNumber`
- `getBoolean`
- `getUrl`
- `getUrlList`

这些函数的作用不是“简化代码”，而是让错误尽量早暴露。

例如：

- `JWT_SECRET` 没配，启动直接失败
- `DB_PORT=abc`，启动直接失败
- `FRONTEND_URL` 不是合法 URL，启动直接失败
- `COOKIE_SECURE` 不是 `true/false`，启动直接失败

这种策略叫 fail-fast，比把错误留到运行期更适合服务端项目。

---

## 7. 安全性改进说明

本次改动里最重要的安全改进有两项：

### 7.1 移除源码中的数据库凭据

旧的 `config.js` 中直接保存了数据库账号密码，这种方式存在以下风险：

- 凭据会进入版本库历史
- 代码泄露即等于凭据泄露
- 不利于不同环境使用不同数据库账户

现在数据库配置全部改为环境变量输入。

### 7.2 JWT 与 Cookie 策略统一归口

JWT 密钥、refresh 密钥、过期时间、Cookie 策略都改为由配置中心统一提供。

这样后续若要做：

- token 时长调整
- 跨域部署
- HTTPS 场景切换
- 反向代理后 Cookie 策略调整

都不需要去业务代码里找散落配置。

---

## 8. 本次改动后的维护建议

后续继续维护时，建议遵循以下规则：

- 新增环境变量时，优先修改 `config/index.js`
- 不要在业务文件中直接新增 `process.env.xxx`
- 新增变量时同步更新 `.env.example`
- 中大型改动同步更新 `CHANGELOG.md`
- 中大型改动同步在 `doc/` 目录新增一份日期说明文档

如果后续出现以下场景，也建议继续沿用这套思路：

- 增加 Redis 配置
- 增加邮件服务配置
- 增加对象存储配置
- 增加第三方 API Key
- 增加多环境数据库连接

统一由配置中心管理，不要回退到“哪里用到哪里读环境变量”的模式。

## 9. 后续可继续优化的方向

这次改动已经建立了基础结构，但后续还可以继续优化：

### 9.1 引入配置 schema 库

目前配置校验是手写的，已经够用，但如果后续变量继续增加，可以考虑引入更专业的 schema 校验库，例如：

- `zod`
- `envalid`
- `joi`

这样可以得到更强的类型约束和更清晰的错误提示。

### 9.2 按环境拆分文档

如果未来部署方式变复杂，可以继续补充：

- 本地开发环境变量说明
- Docker 部署环境变量说明
- 生产环境配置约定

### 9.3 增加启动自检日志

可以在启动阶段输出一份脱敏后的配置摘要，例如：

- 当前环境
- 当前端口
- 当前允许的 CORS 来源
- 数据库目标主机

注意不要打印密码或密钥。

---

## 10. 总结

本次改动的本质，不只是“把 `.env` 读法改了一下”，而是完成了这几件更关键的事情：

- 建立统一配置中心
- 建立启动期校验机制
- 移除源码硬编码敏感信息
- 建立配置模板文件
- 建立 changelog 和技术说明文档

这套结构后续可以作为项目配置管理的基础规范继续沿用。
