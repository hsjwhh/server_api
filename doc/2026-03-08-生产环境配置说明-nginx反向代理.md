# 2026-03-08 生产环境配置说明（Nginx 反向代理）

## 1. 当前生产部署结构

当前讨论确认的线上访问结构如下：

- 前端对外地址：`https://app.example.com`
- 后端对外地址：`https://app.example.com/api`
- Nginx 对外提供 HTTPS
- Nginx 将 `/api` 请求反向代理到后端 Node 服务
- 后端 Node 服务运行在容器或内网服务中，例如 `http://server_api:3000`

这意味着：

- 对浏览器而言，前端和后端是同一个 origin
- API 虽然走的是 `/api` 路径，但本质上仍属于 `https://app.example.com`
- 这不是“跨域不同站点”场景，而是“同域路径转发”场景

---

## 2. 生产环境配置的核心原则

在这种部署方式下，环境变量填写时必须区分两类地址：

### 2.1 浏览器访问地址

这是用户在浏览器里真正访问到的地址，例如：

- `https://app.example.com`

这一类地址用于：

- `FRONTEND_URL`
- `CORS_ORIGINS`
- Cookie 策略判断

### 2.2 内网服务地址

这是 Nginx 反向代理到的后端服务地址，例如：

- `http://server_api:3000`
- `http://127.0.0.1:3000`
- `http://localhost:3000`

这一类地址只应该出现在：

- Nginx upstream 配置
- Docker 网络内部通信

不应该写进：

- `FRONTEND_URL`
- `CORS_ORIGINS`
- 前端构建时对外 API 地址配置

---

## 3. 为什么 `FRONTEND_URL` 不能写成 `/api` 地址

生产环境中，曾经容易产生一个误区：

- 前端地址是 `https://app.example.com`
- API 地址是 `https://app.example.com/api`
- 于是想把 `FRONTEND_URL` 写成 `https://app.example.com/api`

这是不对的。

原因是 CORS 判断依据不是完整 URL，而是 origin。

origin 只包含：

- 协议
- 域名
- 端口

不包含路径。

因此：

- `https://app.example.com`
- `https://app.example.com/api`

它们的 origin 都是：

- `https://app.example.com`

所以：

- `FRONTEND_URL` 应填写 `https://app.example.com`
- `CORS_ORIGINS` 应填写 `https://app.example.com`

而不是 `/api` 路径。

---

## 4. 生产环境推荐 env 配置

根据当前部署结构，推荐的生产环境变量如下：

```env
NODE_ENV=production
PORT=3000

FRONTEND_URL=https://app.example.com
CORS_ORIGINS=https://app.example.com

JWT_SECRET=替换为生产环境正式密钥
JWT_EXPIRES_IN=15m
JWT_REFRESH_SECRET=替换为生产环境正式刷新密钥
JWT_REFRESH_EXPIRES_IN=7d

COOKIE_SECURE=true
COOKIE_SAME_SITE=Lax
COOKIE_MAX_AGE_MS=604800000
COOKIE_DOMAIN=

DB_HOST=你的数据库地址
DB_PORT=3306
DB_USER=你的数据库用户名
DB_PASSWORD=你的数据库密码
DB_NAME=server_info
DB_QUEUE_LIMIT=0
```

## 5. 各生产配置项说明

### 5.1 `NODE_ENV=production`

表示当前运行环境为生产环境。

项目中的配置中心会据此决定一些默认策略，例如：

- 默认启用更严格的 Cookie 安全策略

生产环境必须显式填写为：

```env
NODE_ENV=production
```

### 5.2 `PORT=3000`

这是 Node 服务在容器或内网中实际监听的端口。

Nginx 对外暴露 80/443 并不影响 Node 进程内部监听 3000。

所以：

- Nginx 监听 443
- Node 监听 3000

这是正常的。

### 5.3 `FRONTEND_URL=https://app.example.com`

这个值表示前端页面真实访问地址。

它应该始终填写浏览器地址栏中的前端地址，而不是：

- Nginx upstream 地址
- Docker 容器名
- `localhost`
- `/api` 路径

正确写法：

```env
FRONTEND_URL=https://app.example.com
```

### 5.4 `CORS_ORIGINS=https://app.example.com`

这个值表示允许访问后端的前端 origin 白名单。

虽然当前架构实际属于同域路径转发，理论上多数情况下不会遇到传统跨域问题，但后端 CORS 配置仍应保持明确。

当前推荐写法：

```env
CORS_ORIGINS=https://app.example.com
```

如果未来增加第二个前端域名，可以用逗号分隔：

```env
CORS_ORIGINS=https://app.example.com,https://admin.example.com
```

### 5.5 `COOKIE_SECURE=true`

当前生产环境前端对外访问为 HTTPS，因此 Cookie 应该只在 HTTPS 下发送：

```env
COOKIE_SECURE=true
```

这项在生产环境基本应固定为 `true`。

### 5.6 `COOKIE_SAME_SITE=Lax`

当前场景是：

- 前端和后端同域
- API 通过 `/api` 反向代理

这种情况下通常使用：

```env
COOKIE_SAME_SITE=Lax
```

即可满足需求。

只有在明显跨站并且必须携带 Cookie 时，才通常考虑：

```env
COOKIE_SAME_SITE=None
COOKIE_SECURE=true
```

当前架构不需要这样配置。

### 5.7 `COOKIE_DOMAIN=`

当前推荐留空。

原因：

- 当前只使用 `app.example.com`
- 留空时浏览器会按当前响应域设置 Cookie
- 可减少因 domain 配置错误导致 Cookie 不生效的问题

如果未来要在多个子域共享 Cookie，再考虑显式配置，例如：

```env
COOKIE_DOMAIN=.example.com
```

但在当前架构下，不建议提前复杂化。

### 5.8 JWT 配置

生产环境中这几项必须替换成真实值：

- `JWT_SECRET`
- `JWT_REFRESH_SECRET`

要求：

- 必须足够长
- 必须足够随机
- Access Token 与 Refresh Token 不要共用同一密钥
- 不要把真实值写回仓库

### 5.9 数据库配置

数据库配置按实际部署填写。

如果数据库与服务在同一个 Docker 网络中，常见写法是：

```env
DB_HOST=mysql
```

这里的 `mysql` 指 docker compose 中数据库服务名，而不是公网地址。

如果数据库在独立服务器，则填写对应地址即可。

### 5.10 `TRUST_PROXY` 配置

当 Node 服务运行在 Nginx 或负载均衡器之后时，必须通过该项告诉 Express 信任代理服务器传来的真实 IP。

- **作用**：解决 `express-rate-limit` 报错以及获取到错误的客户端 IP 问题。
- **推荐值**：
  - 生产环境（有 Nginx）：`TRUST_PROXY=true`
  - 开发环境（直连）：`TRUST_PROXY=false`

## 6. Nginx 额外配置建议

为了支持附件（如图片）上传，必须在 Nginx 的 `http` 或 `server` 块中放宽上传限制：

```nginx
# 允许最大上传 10MB 的文件（需与后端 Multer 限制保持一致）
client_max_body_size 10M;
```

如果未配置此项，上传大于 1MB 的文件时 Nginx 会直接返回 `413 Payload Too Large`。

## 7. Docker 部署时 env 的使用方式

当前项目已经支持通过容器环境变量直接启动，不依赖镜像内必须存在 `.env` 文件。

这意味着生产环境更推荐：

- 用 Docker Compose 的 `env_file`
- 或用 `environment`
- 或由部署系统直接注入环境变量

而不是把生产 `.env` 打包进镜像。

建议理解为：

- Docker 负责读取 `env_file`
- Docker 将变量注入容器进程环境
- Node 服务从 `process.env` 中读取这些值

也就是说，后端服务不是“自己去读 Docker 的 env_file 文件”，而是“读取 Docker 已经注入好的环境变量”。

---

## 7. `.env` 与生产配置文件的建议命名

如果是本地开发，建议使用：

- `.env`
- `.env.local`

如果是生产环境，建议：

- 不把真实生产配置文件提交到仓库
- 由运维或部署系统单独维护

例如可以在服务器上保留：

- `.env.production.local`

或者更常见的是：

- 直接使用 Docker Compose 的 `env_file`
- 或平台环境变量注入

仓库中只保留：

- `.env.example`

作为模板。

---

## 8. Nginx 反向代理配置理解

这里再明确一次地址关系：

- 浏览器访问：`https://app.example.com`
- 浏览器请求 API：`https://app.example.com/api`
- Nginx 转发目标：`http://server_api:3000`

所以：

- 前端看到的是 `https://app.example.com`
- 后端实际监听的是 `3000`
- `/api` 是 Nginx 对外暴露的路径前缀
- 不应把 Nginx 内网目标写进前端 origin 配置

## 9. 当前架构下的最终填写建议

当前线上结构下，建议最终按下面这份为准：

```env
NODE_ENV=production
PORT=3000

FRONTEND_URL=https://app.example.com
CORS_ORIGINS=https://app.example.com

JWT_SECRET=你的正式密钥
JWT_EXPIRES_IN=15m
JWT_REFRESH_SECRET=你的正式刷新密钥
JWT_REFRESH_EXPIRES_IN=7d

COOKIE_SECURE=true
COOKIE_SAME_SITE=Lax
COOKIE_MAX_AGE_MS=604800000
COOKIE_DOMAIN=

DB_HOST=你的数据库地址或容器服务名
DB_PORT=3306
DB_USER=你的数据库用户名
DB_PASSWORD=你的数据库密码
DB_NAME=server_info
DB_QUEUE_LIMIT=0
```

## 10. 后续维护建议

后续如果部署结构变化，例如：

- 前后端拆成不同子域
- 前端走 CDN
- API 独立成 `api.example.com`
- 接入 Cloudflare Zero Trust

则优先重新检查这几项：

- `FRONTEND_URL`
- `CORS_ORIGINS`
- `COOKIE_SECURE`
- `COOKIE_SAME_SITE`
- `COOKIE_DOMAIN`

不要只改 Nginx，而忽略后端环境变量配置。

当前这一版说明，适用于：

- 前端和后端同域
- API 通过 `/api` 反向代理
- Nginx 对外提供 HTTPS
- 后端通过 Docker 或内网服务运行
